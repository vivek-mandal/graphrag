# MSCI Knowledge Graph RAG — Parsing & Chunking

Documents are parsed from raw SEC EDGAR HTML filings into structured, metadata-rich text chunks ready for embedding and graph extraction.

---

## Project Overview

**Goal**: A system that answers questions about MSCI in natural language, using a knowledge graph built from MSCI's official SEC filings.

**Documents used:**

| File | Type | Period | Source |
|---|---|---|---|

## What's Inside the Raw Files

The filings are in **iXBRL format** (inline XBRL) — standard HTML with embedded XBRL machine-readable tags, generated by Workiva. This means a single `.htm` file contains two layers:

| Layer | Content | Size |
|---|---|---|
| Visible HTML | Actual document text, tables, images | ~394K chars (10-K) |
| Hidden XBRL metadata | Machine-readable taxonomy tags, entity IDs, date contexts | ~24K chars of garbage |

**From `inspect.ipynb` element counts (10-K):**

| Element | Count |
|---|---|
| Tables | 85 |
| Images | 2 |
| SVGs / Canvases | 0 / 0 |
| `<hr>` page breaks | 105 |
| `<div>` | 2,395 |
| `<span>` | 6,127 |

## Parsing Pipeline (`parse.py`)

### Step 1 — Strip Workiva/XBRL noise

Three types of noise at the top of each file, stripped before parsing:

```
<?xml version='1.0' encoding='ASCII'?>          ← XML processing instruction
<!--XBRL Document Created with the Workiva Platform-->  ← HTML comments
<div style="display:none">...</div>             ← 24K chars of XBRL metadata
```

The hidden `<div>` alone contains 24K chars of machine-readable garbage:
```
00014081982025FYFALSEP3Yhttp://fasb.org/us-gaap/2025#OtherAccruedLiabilities
Currentiso4217:USDxbrli:sharesmsci:segment00014081982025-01-012025-12-31...
```
If not stripped, this ends up embedded in chunks and corrupts retrieval.

### Step 2 — Build section map

All 23 document sections are detected by finding bold `font-weight:700` `<span>` tags matching `Item X.` patterns. Two HTML patterns are handled:

**Pattern A** (18 sections) — number and title in separate sibling spans:
```html
<span style="font-weight:700">Item 1.</span>
<span>Business</span>
```

**Pattern B** (4 sections: 1A, 6, 7, 7A) — title embedded in same span with `\xa0`:
```html
<span style="font-weight:700">Item 7.    Management's Discussion and Analysis...</span>
```

**Pattern C** (Item 1A specifically) — number has no period, period is a separate sibling:
```html
<span style="font-weight:700">Item 1A</span>
<span>.</span>
<span>Risk Factors</span>
```

All 23 sections detected reliably with no fallback needed.

### Step 3 — Walk DOM in document order

The DOM is walked top-down tracking three things simultaneously:

- **Page number** — incremented on every `<hr>` tag (105 page breaks in 10-K)
- **Current section** — updated when a bold Item span is encountered
- **Content type** — `table` or `text`

Tables are converted to pipe-delimited text at this stage (see below). All other visible text nodes are yielded as plain text blocks.

### Step 4 — Table conversion

**Problem**: Plain `get_text()` on a financial table destroys structure:
```
Index Recurring 957,897 882,367 8.6
```
The LLM cannot tell which number belongs to which year.

**Fix**: Convert each table to pipe-delimited rows:
```
[TABLE]
Years Ended | December 31, 2025 | December 31, 2024 | Change
Index Recurring subscriptions | $957,897 | $882,367 | 8.6%
Index Asset-based fees | $770,670 | $657,501 | 17.2%
[/TABLE]
```
Column headers stay attached to values. The LLM can now correctly extract "Index revenue was $957M in 2025".

### Step 5 — Text cleaning

Applied to every passage before chunking:

| Issue | Fix |
|---|---|
| `\xa0` non-breaking spaces | Replace with regular space |
| Smart quotes `\u2018` `\u2019` `\u201c` `\u201d` | Replace with `'` and `"` |
| Em/en dashes `\u2014` `\u2013` | Replace with `-` |
| `\ufffd` replacement chars (encoding errors) | Remove |
| `_{5,}` footnote separator lines | Remove (keep footnote text that follows) |
| Excessive whitespace | Collapse to single space / double newline |

### Step 6 — Chunking

```python
RecursiveCharacterTextSplitter(
    chunk_size=2000,
    chunk_overlap=200,
    separators=["\n\n", "\n", ". ", " "]
)
```

**Why 2000 chars (not 1000)?**
- At 1000: ~430 chunks from the 10-K alone — more Neo4j nodes, higher embedding cost
- At 2000: ~416 chunks total — each chunk has more context for entity extraction
- The 14 large tables (max 2,922 chars in 10-K) now mostly fit in one chunk
- 2000 chars ≈ 500 tokens — well within any LLM context window

**Why `RecursiveCharacterTextSplitter`?**
Tries to break at natural boundaries (`\n\n` → `\n` → `. ` → ` `) before hard-cutting at 2000 chars. Avoids splitting mid-sentence.

**Why 200 overlap?**
10% overlap ensures entities that span a paragraph boundary aren't lost at chunk edges.

### Step 7 — Metadata attachment

Each chunk gets 8 metadata fields:

```json
{
    "chunk_index": 42,
    "page_content": "Index revenues increased 8.6%...",
    "source": "https://www.sec.gov/Archives/edgar/data/1408198/.../msci-20251231.htm",
    "filename": "msci_10k_fy2025.htm",
    "filing_type": "10-K",
    "period": "FY2025",
    "page_number": 41,
    "section": "Item 7 - Management's Discussion and Analysis...",
    "chunk_type": "table"
}
```

| Field | Value | Why useful |
|---|---|---|
| `source` | SEC EDGAR URL | Direct link back to source document |
| `filename` | Local filename | Local file operations |
| `filing_type` | `10-K` or `10-Q` | Filter by report type |
| `period` | `FY2025` or `Q1 2026` | Time-aware queries |
| `page_number` | Integer | Traceability to physical page |
| `section` | `Item 1A - Risk Factors` | Pre-filter before vector search |
| `chunk_type` | `text` or `table` | Know if chunk contains financial data |
| `chunk_index` | Integer | Ordering and debugging |

### Step 8 — Save as JSONL

One JSON object per line in `data/processed/`. Chosen over a single JSON array because:
- Streamable — process one chunk at a time without loading all 593 into memory
- Greppable — `grep "Item 1A" file.jsonl` works directly
- Standard format for LLM/embedding pipelines

## What Was Deliberately NOT Done

| Skipped | Why |
|---|---|
| OCR on images | Both images are logo + decorative JPG — no financial data |
| Multimodal LLM for images | Zero financial content in images — unnecessary cost |
| CUSIP / ISIN extraction | Not present in 10-K/10-Q filings (those live in prospectuses) |
| Section-aware hard splits | `RecursiveCharacterTextSplitter` + section metadata achieves the same result without rigid boundaries |
| PDF parsing | SEC EDGAR hosts native HTML — cleaner text, no PDF layout artifacts |
| Single JSON array | JSONL is streamable and greppable; no reason to load all 593 chunks at once |
